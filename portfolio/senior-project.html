Python

About

Python is a high-level programming language with general-use. Knowing that I will need to work with Python at some point this summer, I decided to research advanced Python topics.



Point

PYTHON

  • Interpreted language: no compilation and linking required
  • Extensible language
  • Code Style:
    • Use 4-space indentation (no tabs)
    • Wrap lines at max 79 characters
    • Blank lines to separate functions, classes, larger blocks of code inside function
    • Put comments on their own line whenever possible
    • Use docstrings
    • Naming Conventions:
      • Classes: UpperCamelCase
      • Functions/Methods: lowercase_with_underscores
    • Always use self as the first method argument name

CONTROL FLOWS

  • if Statements
    • Can have optional elif and else part(s)
    • Substitute for switch/case statements found in other languages
  • fpr Statements:
    • Iterates over items in a sequence
    • If iterated sequence needs to be modified, first create a copy via list[:]
  • range() Function:
    • Generates arithmetic progressions
    • range(start, stop, step)
      • start: 0 if parameter not supplied
      • stop: always supplied
      • step: 1 if parameter not supplied
    • Behaves like a list, but isn't one - it's an object that returns the sequence, but does not create the list (saves space)
    • Example Usage: list(range(x)) -> creates a list
  • break and continue Statements:
    • break: breaks out of the innermost enclosing for or while loop
    • continue: continues with the next iteration
  • else Clauses on Loops:
    • Executed when the for loop terminates after iterating through a list
    • Executed when the while loop condition becomes false
    • Not executed when break is executed
  • pass Statements:
    • Does nothing
    • Used when required by syntax, but not by action
    • Used as a placeholder
  • Defining Functions:
    • def funcName(parameters):
      • When executed, a new symbol table for local variables (specific to the function) is introduced
      • When a variable is referenced, it first searches the local symbol table, then that of the enclosing function, then the global one, then built-in names
      • Parameters/arguments are created where the function was called - the value is an object reference, not the value itself
      • All functions, even those without a return statement, are functions: it returns the value None
    • Keyword Arguments:
      • def funcName(parameter1, parameter2=x)
      • Specify a default value to argument(s)
      • Evaluated at the point of function definition in the defining scope
      • Evaluated only once
    • Sub
      • kwarg=value
      • Must follow positional arguments
      • Order does not matter
      • No argument can be referenced more than once
      • *name: tuple of all positional arguments received
      • **name: final formal parameter; dictionary of all keyword arguments received
    • Arbitrary Argument Lists:
      • Arguments get wrapped up in a tuple
      • Often the last in list of formal parameters
    • Unpacking Argument Lists:
      • When arguments are already in a list/tuple
      • Use * operator to unpack
      • Use ** to unpack a dictionary of keyword arguments
    • Lambda Expressions:
      • lambda vars, listed: expression
      • References variables from the containing scope
      • Can be passed as an argument through a function
    • Documentation Strings:
      • First line: short, concise summary of object's purpose
        • Should not state object's name/type explicitly
        • Begin with a capital letter, end with a period
      • Second line: left blank to visually separate the summary and description
      • Following lines: paragraph(s) explaining object's calling conventions, side effects, etc.
      • Can be referenced via func_name.__doc__
      • White space determined by first non-blank line
    • Function Annotations:
      • Optional metadata information about the types used
      • Can be referenced via func_name.__annotations__
      • Is a dictionary that has no effect
        • Includes positional argument, keyword argument, and return value
        • Ex: {'var_name': <class 'type'>, 'return': <class 'type'>}

DATA STRUCTURES

  • Lists:
    • Methods of List Objects:
      • append(x): add item to end of list
      • extend(iterable): extend list by appending all items from another list
      • insert(i, x): insert an item at a position
      • remove(x): remove the first item in the list whose value is equal to x
      • pop([i]): remove the item at a given position and return it; removes last item if index is not provided
      • clear(): remove all items in the list
      • index(x[, start[, end]]): return index of first item with the value x; raises a ValueError if it does not exist
      • count(x): return number of x's in the list
      • reverse(): reverse the sequence of items in the list
      • copy(): return a shallow copy of the list
    • Lists as Stacks: last-in, first-out
    • Lists as Queues: first-in, first-out
      • Lists are not efficient for queues - all other elements have to shift by one
      • Use collections.deque to implement a queue (has appends/pops from both ends fairly efficient)
    • List Comprehensions: concise method of list creation
    • Nested List Comprehension: any arbitrary expression including another list comprehension
  • del Statement: remove an item from a list given its index (not value)
    • Can be used to delete entire variables
  • Tuples and Sequences: a set of values
    • Use ()
    • Tuples are immutable and often contain multiple types
    • Tuple Packing: t = 123, 321, 'abc'
    • Tuple Unpacking: a, b, c = t
  • Sets: unordered collection with no duplicates
    • Use set() for an empty set - {} creates an empty dictionary
    • Supports union, intersection, difference, symmetric difference
    • Supports set comprehensions
  • Dictionaries: key: value pairs
    • "associative memories" or "associative arrays"
    • Indexed by keys (any immutable type)
    • Supports dict comprehensions
    • Can create via dict() function and keyword arguments (variable name becomes key)
  • Looping Techniques:
    • Loop through key value pair via for k, v in dictionary.items():
    • Loop through sequence - retrieve index and value at the same time via enumerate()
    • Loop through two or more sequences simultaneously via zip()
    • Loop through sequence in reverse via reversed()
    • Loop through sequences in sorted order via sorted()
  • More on Conditions:
    • Comparison Operators:
      • in and not in: check whether a value (does not) occur in a sequence
      • is and is not: compare whether two values are (not) the same object
      • Priority is lower than that of all numerical operators
    • Boolean Operators:
      • and and or
      • not: negates the outcome of a comparison
      • Priority: not has the highest, or has the lowest
  • Comparing Sequences and Other Types:
    • Uses lexicographical ordering: comparison of corresponding items at a given index in order
    • Examples:
      • (1, 2, 3) < (1, 2, 4)
      • 'ABC' < 'C' < 'Pascal' < 'Python'
      • (1, 2, 3, 4) < (1, 2, 4)
      • (1, 2) < (1, 2, -1)
      • (1, 2) == (1.0, 2.0)

K-Nearest Neighbors (KNN)

  • Identifying based on pixels in the proximity
  • A single template cannot be applied to all cases

Linear Classifiers

  • Parametric approach (every pixel value is a parameter)
  • Weighted sum of the pixel value
  • A single template cannot be applied to all cases

Activation Function

  • Sigmoid Function
    • Ïƒ(x) = 1/(1+e-x)
    • Input: (0, 1)
    • Issues:
      • Saturated neurons kill the gradient
      • Output is (0, 1), not centered at 0
      • Exponential calculations require more computational power
  • tanh Activation:
    • Benefit over sigmoid function:
    • Issue:
      • Still kills the gradient
  • Rectified Linear Unit (ReLU):
    • y = max(0, x)
    • All negative values become zero (remove all black)
    • Simple thresholding -> more computationally efficient
    • Issues:
      • Output is not zero-centered
      • No gradient at x<0 (leaky ReLU and parametric ReLU can mitigate this)
      • No gradient at x=0

Weight Initialization:

  • All Zeros:
    • Often bad idea - neurons generate the same output
  • Gaussian Random Variables:
    • Works for shallow networks, not for deep networks
  • Xavier Initialization:
    • Variance of Gaussian distribution of weights depends on number of layer inputs
    • Recommended: sqrt(inputs)
      • np.random.randn(n_in, n_out)*sqrt(2/n_in)

Convolutional Neural Networks (CNNs or ConvNets):

  • Convolution Layer:
    • image height * image width * 3 deep (RGB)
    • Filter: black/cuboid with smaller height and width, but same depth
    • WTX + b = new value, the weighted sum of the pixels
    • Each filter is an activation map - n filters create n stacked activation maps
    • Output size = (N-F)/S + 1
  • Pooling Layer:
    • Taking regions of the image and combining it to one pixel, via mean, maximum, etc.
  • Fully Connected Layer:
    • Each pixel = separate neuron

Final Report: Developing a Physical Complement to the CLOUDES Web Service for the Visually Impaired

Introduction

Over the past decade, the Internet has made countless services available to millions of people. However, this virtual world remains largely inaccessible to the visually impaired, those who have a functionally limited eyesight as a consequence of a disease or disorder[1].


One such service is CLOUDES, a web service for creating simulations. Its visual-based interface makes it user-friendly, promoting use by non-professionals and students. In order to make it accessible to the visually impaired, physical representations of the digital interface were explored. Tactile components would enable the visually impaired to create a physical model, which can be translated into the CLOUDES web service via computer vision processing of the image.



Discussion

In CLOUDES, several different types of elements can be added to the simulation model such as Arrival, Queue, Process, and Exit. The user can then connect each element together to create the correct sequence of the procedure. The physical model sought to maintain this through unique pieces for each element, which can be put together like a puzzle. A previous design used circular-shaped tabs to connect pieces, which allowed flexibility in motion. However, a trapezoidal shape was used to preserve the rigid uniformity in the constructed model.


As each element in the CLOUDES software could be parameterized, characterizing the behavior of each step in the process, chips were designed to be placed onto each piece. Each piece had a unique shape corresponding to its function: hexagons corresponded to distribution specifiers, triangles corresponded to actions, etc. Simple geometric shapes were used for the chips as a study done by researchers from the Transworld Institute of Technology in Taiwan had found them to be more easily identified than complex icons by touch[2]. Each chip also had a unique embossing of a letter or number to further specify its function. A braille alternative was developed as well. Engravings were originally considered but opted against as most prints use embossing, making it more familiar to the visually impaired.


To further assist in braille recognition, standard braille dimensions from the National Library Service for the Blind and Physically Handicappped were observed[3]. The final design used sizes which scaled to be 1.5 times larger than standard to enhance computer vision performance and to maintain relative distances between dots. The dot diameter was increased from 1.6 mm to 2.4 mm and dot base spacing was increased to 3.6 mm. However, to prevent potentially rough braille, the dot height remained unscaled at 0.9 mm. Additionally, these dots were made of hemispheres rather than cylinders to also reduce roughness.


The physical design was also flat to reduce as much use of materials. It also had rounded corners to be more gentle to the touch.


To enable a computer to recognize the pieces and chips from an image, OpenCV, an open source package with tools for real-time computer vision, was explored. The idea was to identify each piece of the model using its unique shape and color, then to identify each chip and its corresponding braille or character embossing. This data could then be converted into the CLOUDES web service to run the simulation.


A morphological transformation on the image best created the outline of the shapes. However, due to little contrast in color between the embossings and the chips itself, it was challenging to highlight the key features of the model. Additionally, the wood grains of the desk in the background and even the layers of 3D-printed material added to noise to the image.


Although character recognition is widely accessible, braille recognition remains somewhat unexplored. A paper from King Saud University outlines the image transformation steps that has yielded a more efficient recognition of braille cells[4]. The process involves first rotating the image to align the braille vertically and horizontally, detecting each dot, and then recognizing each cell. Each cell could be identified by segmenting then constructing lines in regular intervals according to a scale. The intersection of the dots and lines allow identification of each cell.


Identifying each braille character requires another process. In a paper from the Proceedings of the 17th International Conference on Pattern Recognition, such a system of optical recognition was proposed[5]. It would identify each column of braille dots, also known as as a half character. Though the paper recognizes room for further optimization, it claims that the current system achieves a 99% accuracy rate in recognition.



Conclusion

Though not used in the testing samples, the final physical model should use different colors for each set of pieces and chips to enhance recognition accuracy by computer vision. Furthermore, as the pieces were constructed from a combination of geometric shapes, pictures of the pieces need to be taken under different conditions to create a dataset train the program to recognize each one.
